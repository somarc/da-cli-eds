Calibration notes

Postmortems are historical signal, not current site truth.

da-cli improves when agent reviews, implementation misses, and launch-readiness gaps are preserved as evidence. The point is not to freeze stale project claims. The point is to convert hard lessons into better primitives, audits, pipelines, jobs, and docs.

Read With Date Awareness

Calibration notes can reference a specific site, repository, or migration moment. Those details must be re-verified before they become current backlog. Their durable value is the pattern they expose: what the CLI failed to make obvious, what the audit should have caught, and where an agent had to improvise.

How Feedback Becomes Product Work

Primitive gap

The CLI cannot inspect or mutate the thing directly.

Promote repeated workarounds into named commands with structured output, dry-run behavior, and test coverage.

Pipeline gap

The workflow is expressible, but not ergonomic.

Turn repeated sequences into reviewable YAML DAGs with explicit dependencies, approvals, retries, and reports.

Job gap

The work is long-running or mutation-heavy.

Use durable state, cancellation, verification, ownership claims, and honest partial-failure reporting.

Audit gap

The review missed launch-readiness risk.

Move beyond block inventory into runtime safety, metadata, query/index viability, redirects, analytics, accessibility, link health, and content depth.

Docs gap

The model is right, but hard to understand quickly.

Clarify the distinction between primitives, pipeline DAGs, durable jobs, and agent operating loops.

Consensus Review Model

  1. Collect raw findings from specialized and general agents without treating any single agent as truth.
  2. Normalize each finding as a bug, primitive gap, pipeline gap, job gap, audit gap, docs gap, or speculative idea.
  3. Require concrete evidence: file path, command, failing case, missing behavior, browser observation, or reproducible workflow.
  4. Prefer fixes that improve agent trust in DA and EDS mutations, especially where the operation writes remote state or changes launch behavior.
  5. Make the repo decide with tests, docs, and small reviewable PRs.
