Start with intent, but make the site content-driven first.

What “Loop” Means Here

A loop is a small operator program: it prompts the agent by running known commands, reads structured output, decides whether the site is ready, and either stops or runs the next command. In da-cli, that loop is not magic autonomy. It is a reviewable DAG of DA and EDS primitives with dry-run defaults, approval gates, public preview evidence, and repeatable diagnostics.

The altitude shift

The engineer still decides what should exist, what risk is acceptable, and when to ship. The work moves up a level: from manually driving every content write, preview, audit, and repair, to writing the workflow that drives those actions and records what happened.

That is why da-cli invests in JSON envelopes, route resolution, site doctor output, pipeline YAML, durable jobs, and quality gates. The model becomes a subroutine. The loop owns the boring repetition. The human owns judgment.

while not ready:
  observe = da status / site doctor / audit
  plan = choose next safe primitive
  if plan mutates remote state:
    require approval and root --commit
  run plan
  verify preview, codebus, routes, and browser state
  record friction for the next CLI primitive

The loop is plumbing. The skill is the asset.

The loop matters because it keeps the work moving, but the durable leverage is in the sharp, tested, named things it can call. In da-cli, those assets are commands and skills like da status, da resolve, da site doctor --agent, da content put-tree, da preview explain, da audit contracts, da code sync block, and project-specific pipeline nodes.

A loop with no reusable primitives is just repetition around an unknown actor. A loop calling a library of known DA and EDS skills is a system that compounds: every hard workflow can become a named primitive, and every repeated workflow can become a tested skill.

Loop authoring is the product surface.

The useful move is not to rank modes of work. It is to make the loop explicit enough that an agent can run it, a human can inspect it, and the CLI can improve from the friction it records. A good loop takes an objective, decomposes the work, chooses the right role, command, reusable skill, or pipeline step, orders the dependencies, executes, reviews against the original goal, gates before shipping, and reports only what matters.

Operating Model

Orient with the CLI first

Use the command surface as the interface contract.

Start with da status, da resolve, da site doctor --agent, da preview explain, and the public command reference. The agent should know what da-cli can do before inventing side scripts.

Prepare the site

Make first-run readiness explicit.

Clone the boilerplate repo, verify AEM Sync is installed, confirm or create fstab.yaml, and treat DA mount readiness as a gate before upload.

Author content

Let pages and messages lead.

Create the information architecture and semantic source documents under content/. Blocks come after the content shape is visible.

Validate objectively

Use gates, not vibes.

Upload with content put-tree, preview with root --commit, audit with quality gates, reconcile gaps, and use browser evidence for runtime errors.

Close the loop

Decide whether another turn is justified.

A useful loop has a stop condition: all checks pass, the remaining risk is named, or the next action requires human product judgment. It should not keep prompting because it can.

Recommended Starting Flow

  1. Receive the repo URL for a site created from the boilerplate, with AEM Sync installed.
  2. Clone the repo locally and inspect the existing project shape before adding files.
  3. Verify fstab.yaml points / to the intended DA content mount.
  4. Draft the site map, core narratives, shared navigation, footer, metadata, and page content under content/.
  5. Run da content put-tree content as a dry run, then upload with root --commit.
  6. Run da --commit preview tree / so shared content such as nav, footer, and fragments exists before judging pages.
  7. Add or refine blocks only where the content requires a richer primitive such as cards, tabs, comparison tables, calculators, forms, search, or related articles.
  8. Run quality gates, browser-console validation, and reconciliation. Record every workaround in a friction log so the CLI can improve.

Generalized Seed Prompt

Use this prompt at the start of an agentic session when the goal is to operate on a DA-backed EDS site with da-cli.

We are using da-cli to turn a concise goal into a coordinated DA-backed Edge Delivery Services workflow.

Goal contract:

- Convert the objective into a visible work plan.
- Decompose the work into content, code, preview, audit, and release concerns.
- Assign each concern to the right role, command, reusable skill, or pipeline step.
- Order dependencies so content, blocks, preview, and validation happen in the right sequence.
- Gate before shipping, and report only what matters.

Before making changes, orient with:

1. da status --format json
2. da resolve  --format json
3. da site doctor --agent --format json
4. da preview explain  --format json
5. The public Commands, Quick Reference, and Pipeline docs

da-cli usage contract:

- Use da-cli as the primary orchestration surface.
- Prefer named commands over ad hoc scripts: status, resolve, site doctor, content put-tree, preview, audit, reconcile, code sync, and pipeline quality gates.
- Treat dry-run output as planning evidence.
- Require root --commit for remote mutations.
- Use --branch for code-side isolation and --draft for content-side isolation.
- Capture structured output with --format json when another agent, loop, or pipeline step needs to read the result.
- When da-cli lacks a needed primitive, record the gap instead of silently hiding the workaround.
- da-cli does not replace product judgment or agent coordination; it gives those loops inspectable primitives, structured outputs, and safety gates.

Pipeline DAG contract:

- Promote repeated work into a pipeline DAG.
- Make dependencies explicit: prepare, author, upload, preview, audit, repair, validate, report.
- Put approval gates before remote writes and release actions.
- Keep retries bounded and evidence-driven.
- Stop the loop when checks pass, risk is named, or the next decision is product judgment.

Important development model:

- Treat EDS as content-driven first.
- Start by authoring the site information architecture and page content under content/.
- Verify fstab.yaml and DA mount readiness before uploading content.
- Use block primitives only after the content model makes the need clear.
- Prefer semantic content and progressive enhancement over decorative block-first layout.
- Use da content put-tree for DA upload.
- Use da --commit preview tree or da --commit preview page  for preview rebuilds, and da preview status for freshness checks.
- Use da pipeline quality-gate, da audit contracts --verify-code, and da site reconcile to audit and improve the result.
- If local validation is needed, use da up as the local EDS server substrate.
- Browser-console validation means opening the local or preview URL in a real browser and checking console, page, and network errors; da up alone is not enough.

Goal:

Create a complete, high-quality DA-backed EDS workflow from a concise business intent. The work should exercise reusable primitives only where they naturally fit: navigation, metadata, hero, cards, comparison tables, tabs, forms, search, related articles, calculators or interactive tools, fragments, footer, and quality gates.

While working, keep a friction log with:

- What the agent tried
- Which da-cli command helped
- Where the CLI was missing a primitive
- Any manual workaround used
- Suggested da-cli improvement

Do not start by inventing blocks. Start by designing the content map, then author content, then add or refine blocks where the content demands richer behavior.

What This Workflow Tests

The important question is not whether an agent can hand-build a nice site. The question is whether da-cli gives the agent enough safe, composable primitives to move from repo readiness to content upload, preview, audit, reconciliation, and runtime validation without hidden manual work. The loop narrative is only honest when every step is inspectable, every mutation is gated, and every retry has evidence.
