CLI decision guide

Use DA CLI for operations and agent-local checks. Use AEM CLI for full local development.

@somarc/da-cli is the operational CLI for DA-backed EDS sites: content administration, preview, publish, deploy, route ownership, site diagnostics, audits, jobs, pipelines, agent-readable JSON, and da up for deterministic local/proxied checks. Adobe's AEM CLI lineage remains the right tool for the full human development loop: aem up, local proxying, importer UI work, live reload, TLS, and browser ergonomics.

Fast Read

The tools overlap around DA content, but their center of gravity is different. Treat da as the release, diagnostics, automation, and deterministic local-check surface. Treat aem or hlx as the full local development surface.

@somarc/da-cli

Binary
da
Primary use
Operate DA-backed EDS sites across auth, config, content, preview, publish, deploy, route, index, audit, migrate, pipeline, job, code, block, commerce, site diagnostics, and up.
Safety model
Write operations are dry-run by default and require root --commit. Output formats include table, JSON, and Markdown.
Best fit
Remote operations, release checks, content administration, automation, CI, agent workflows, and deterministic local/proxied route checks.

@adobe/aem-cli / hlx lineage

Binaries
aem and compatibility alias hlx
Primary use
Run the full local AEM/EDS development server, run the importer server, and manage a local DA content workspace.
Safety model
Content workspace commands mirror Git concepts. The dev server provides local rendering, proxying, TLS, live reload, and browser log forwarding.
Best fit
Local authoring, block development, importer UI work, and browser-runtime debugging.

Use Both

Pattern
Use da up for agent-friendly local/proxied checks. Use aem up for the full browser-oriented development loop. Then validate, preview, publish, and diagnose with DA CLI.
Hand-off
Use da status, da resolve, and da site doctor --agent when work moves from local editing to remote site state.
Best fit
Teams that need a clean split between local development ergonomics and production-oriented operations.

Decision Guide

Use da-cli for operations

Choose it for preview, publish, deploy, DA content administration, route ownership, code-bus checks, index validation, site diagnostics, audits, jobs, pipelines, and migration automation.

Use Adobe AEM CLI for local dev

Choose it for the full aem up experience: local proxy behavior, live reload, TLS, local content rendering, forwarded browser logs, and the importer server.

Use both in a mature workflow

Run AEM CLI lineage while developing locally, use da up for deterministic local/proxied checks, then use DA CLI for validation, publishing, audit, and diagnostic workflows.

Bottom line: Adobe's CLI is the official local development tool. DA CLI is the broader EDS operations CLI and the source of truth for release checks, route ownership, full DA CRUD, migration jobs, agent-oriented JSON, and destructive actions that need guardrails.
