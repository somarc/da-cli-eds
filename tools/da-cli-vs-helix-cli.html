Deep CLI comparison

DA CLI is the operational control plane, and our AEM CLI fork is the parity bridge.

Adobe's official @adobe/aem-cli, still shipped from the Helix CLI repository, is strong for local development, import workflows, and local DA content checkout. Our @somarc/aem-cli fork carries a focused subset of DA CLI operational surfaces. @somarc/da-cli goes wider: DA Admin API automation, preview and publish orchestration, route ownership, index validation, audits, site diagnostics, migration, code-bus checks, and agent-friendly safety rails.

Product Profiles

The tools overlap around DA content, but their center of gravity is different. Adobe's CLI optimizes the authoring developer loop. The AEM CLI fork selectively imports operational paths from da-cli. da-cli remains the broadest surface for DA, Helix Admin, EDS routing, publishing, validation, and automation.

@somarc/da-cli

Binary

@somarc/aem-cli fork

Binary
aem
Primary use
Bridge local AEM CLI ergonomics with selected da-cli operations: auth, DA content workspace, preview/publish/deploy, route diagnostics, index checks, code-bus sync, and local development commands.
Safety model
Adopts explicit commit gates for publish, deploy, and code-bus mutation while preserving the familiar aem command shape.
Runtime deps
Forked command registry and workflow docs focused on parity mapping rather than full da-cli replacement.

@adobe/aem-cli from helix-cli

Binaries
aem and compatibility alias hlx
Primary use
Run a local AEM/EDS development server, run the importer server, and manage a local DA content workspace.
Safety model
Content workflow mirrors Git concepts with clone, add, commit, status, diff, merge, and push. Local server provides live reload, proxying, TLS, and browser log forwarding.
Runtime deps
Feature-rich Node service stack for dev server, proxying, import UI, shared Helix config, Git helpers, and browser injectables.

Auth parity is already in the fork and should stay aligned with da-cli token behavior.

Decision Guide

Use da-cli for operations

Choose it for preview, publish, deploy, DA content administration, route ownership, code-bus checks, index validation, site diagnostics, audits, and migration automation.

Use the AEM fork for parity pilots

Choose it when an operator wants the familiar aem shape but needs selected da-cli surfaces for auth, content workspace, delivery, route, index, and code-bus work.

Use Adobe AEM CLI for local dev

Choose it for aem up, local proxy behavior, live reload, TLS, local content rendering, forwarded browser logs, and the importer server.

Use both in a mature workflow

Run AEM CLI lineage while developing locally, use the fork where parity exists, then use da-cli for the broader validation, publishing, audit, and diagnostic surface.

Bottom line: Adobe's CLI is the official local development tool. The AEM CLI fork is a practical subset for migrating selected da-cli operations into the aem command shape. da-cli remains the broader EDS operations CLI and the source of truth for gaps like audits, site doctor, full DA CRUD, migration jobs, and destructive route cleanup.
