Deep CLI comparison

DA CLI is the operational control plane Adobe's official CLI does not try to be.

Adobe's official @adobe/aem-cli, still shipped from the Helix CLI repository, is strong for local development, import workflows, and local DA content checkout. @somarc/da-cli goes wider: DA Admin API automation, preview and publish orchestration, route ownership, index validation, audits, site diagnostics, migration, code-bus checks, and agent-friendly safety rails.

Product Profiles

The two tools overlap around DA content, but their center of gravity is different. Adobe's CLI optimizes the authoring developer loop. da-cli optimizes operational work across DA, Helix Admin, EDS routing, publishing, validation, and automation.

Deep Comparison Matrix

This matrix is intentionally practical: it compares what an EDS engineer or automation agent can actually do from the command line today based on the local repositories.

Local development server AEM CLI wins da-cli does not implement a local proxy/dev server. Strong aem up serves local projects, proxies remote content, supports live reload, TLS, browser opening, forwarded browser logs, and local HTML folder mounting. Use Adobe's CLI when the job is browser-local development and asset/runtime simulation.

Partial DA content workspace commands exist, but the documented surface is focused on local checkout and push. da-cli is better for direct DA repository administration and scripts that need exact remote operations.

Not documented The official README does not expose preview orchestration commands. da-cli owns the full source-to-preview promotion loop from CLI automation.

Tie Supports local DA content clone, status, diff, add, commit, push, and merge; aem up also serves local content first. Route ownership and destructive safety da-cli wins route classify, route canonical, route audit, and route clean distinguish contentbus, codebus, hybrid, orphan, and probe-failed routes. Not documented No equivalent route ownership model surfaced in the official CLI documentation. da-cli is safer for cleanup and migration work where DA and code-owned routes can coexist.
Site diagnostics da-cli wins site doctor checks registration, content source type, code-bus visibility, shared DA docs, preview/live state, and optional deep sampling. Not documented No site doctor command in the official README command set. da-cli provides a single diagnostic entry point for account, repo, route, and delivery-state questions.
Search index operations da-cli wins index show, index validate, and index query inspect helix-query.yaml and compare configured fields with live responses. Not documented No index inspection or query command in the official README. da-cli turns query-index validation into a repeatable CLI check.
Quality audits da-cli wins Audits cover semantics, blocks, full-page checks, and contracts. Not documented No comparable audit command group in the official command list. da-cli is better for preflight checks before preview, publish, or migration completion.
Migration tooling da-cli wins Migration commands include import, batch import, job status, and validation. Different strength aem import runs the importer server; it is UI/server oriented rather than remote job automation. Use Adobe's CLI for the importer UI server; use da-cli for scripted migration flows and validation.
Code-bus visibility da-cli wins Code commands cover sync, status, job lookup, sidekick config, and purge. Not documented The official surface does not expose code-bus operational commands. da-cli reaches beyond DA content into EDS code delivery state.
Config model da-cli wins Project and global DA config, environment fallback, branch defaults, source annotation, and per-command overrides are first-class. Focused Environment variables configure local server and global logging; shared Helix config is consumed by the dev server. da-cli is stronger for repeatable multi-repo automation; Adobe's CLI is tuned for local project runtime configuration.
Automation output da-cli wins Global --format table|json|md, --quiet, --verbose, and request IDs make it script and agent friendly. Partial Logs are configurable, but the official README does not present structured output as a cross-command contract. da-cli is easier to compose in CI, agent workflows, reports, and shell scripts.
Authentication ergonomics da-cli wins auth login, logout, status, and token expose explicit token lifecycle management. Internal DA auth support exists under content internals, but the official README does not document standalone auth commands. da-cli is clearer when credentials need to be inspected, renewed, or piped into other tools.

Decision Guide

Use da-cli for operations

Choose it for preview, publish, deploy, DA content administration, route ownership, code-bus checks, index validation, site diagnostics, audits, and migration automation.

Use Adobe AEM CLI for local dev

Choose it for aem up, local proxy behavior, live reload, TLS, local content rendering, forwarded browser logs, and the importer server.

Use both in a mature workflow

Run Adobe's server while developing locally, then use da-cli to validate, preview, publish, audit, and diagnose the DA-backed site before release.

Bottom line: Adobe's CLI is the official local development tool. da-cli is the broader EDS operations CLI. If the goal is best-in-class command-line control over DA-backed sites, da-cli has the deeper surface area and the safer automation model.
